# Operation Journaling & Sync Requirements

In distributed and offline-first systems, simply forwarding patches from one replica to another isn't always enough. Network partitions or prolonged offline periods mean a replica might miss dozens of operations, and broadcasting the entire document state to catch up is often inefficient.

Ama.CRDT provides a **Journaling** architecture. By configuring the DI container to intercept patch generation and application, the library can automatically append every successfully applied or generated `CrdtOperation` to a centralized, append-only log (an `ICrdtOperationJournal`). 

You can then use the `IJournalManager` alongside the `IVersionVectorSyncService` to query exactly which operations a disconnected node is missing and send only those missing bytes.

## 1. Implementing the Journal

To enable journaling, you must first implement the `ICrdtOperationJournal` interface. This serves as an adapter to your preferred database, message queue, or event store.

```csharp
using Ama.CRDT.Models;
using Ama.CRDT.Services.Journaling;
using System.Collections.Generic;
using System.Runtime.CompilerServices;
using System.Threading;
using System.Threading.Tasks;

public class MyDatabaseJournal : ICrdtOperationJournal
{
    public void Append(string documentId, IReadOnlyList<CrdtOperation> operations)
    {
        // Synchronous append (e.g., using a traditional DB driver)
        SaveToDatabase(documentId, operations);
    }

    public async Task AppendAsync(string documentId, IReadOnlyList<CrdtOperation> operations, CancellationToken cancellationToken = default)
    {
        // Asynchronous append
        await SaveToDatabaseAsync(documentId, operations, cancellationToken);
    }

    public async IAsyncEnumerable<JournaledOperation> GetOperationsByRangeAsync(string originReplicaId, long minGlobalClock, long maxGlobalClock, [EnumeratorCancellation] CancellationToken cancellationToken = default)
    {
        // Query operations generated by a specific replica within a contiguous clock range.
        // E.g., SELECT * FROM Ops WHERE Origin = @originReplicaId AND Clock > @min AND Clock <= @max
        await foreach (var op in FetchFromDatabase(originReplicaId, minGlobalClock, maxGlobalClock))
        {
            yield return op;
        }
    }

    public async IAsyncEnumerable<JournaledOperation> GetOperationsByDotsAsync(string originReplicaId, IEnumerable<long> globalClocks, [EnumeratorCancellation] CancellationToken cancellationToken = default)
    {
        // Query specific out-of-order operations (dots)
        // E.g., SELECT * FROM Ops WHERE Origin = @originReplicaId AND Clock IN (@globalClocks)
        await foreach (var op in FetchDotsFromDatabase(originReplicaId, globalClocks))
        {
            yield return op;
        }
    }
}
```

## 2. Registering Journaling Decorators in DI

The library provides decorators (`JournalingApplicatorDecorator` and `JournalingPatcherDecorator`) that seamlessly intercept operations. You register them in your Dependency Injection configuration:

```csharp
using Ama.CRDT.Extensions;
using Ama.CRDT.Services.Decorators;

var builder = WebApplication.CreateBuilder(args);

builder.Services.AddCrdt()
    // 1. Register your custom journal implementation
    .AddCrdtJournaling<MyDatabaseJournal>()
    
    // 2. Decorate the applicator so it saves operations as they are applied
    .AddCrdtApplicatorDecorator<JournalingApplicatorDecorator>()
    
    // 3. Decorate the patcher so it saves operations when generated explicitly
    .AddCrdtPatcherDecorator<JournalingPatcherDecorator>();
```

With this configuration:
- Calling `ICrdtApplicator.ApplyPatchAsync` will automatically trigger `ICrdtOperationJournal.AppendAsync` for any operation that was successfully applied (skipping rejected or duplicate operations).
- Calling `ICrdtPatcher.GeneratePatchAsync` or `.BuildOperationAsync` will similarly push generated operations to the journal.

## 3. Resolving Sync Requirements

When a replica comes back online, it needs to catch up. Ama.CRDT tracks causality using Dotted Version Vectors (`CrdtMetadata.VersionVector` and `SeenExceptions`).

To find out what a replica is missing, you calculate a `ReplicaSyncRequirement` using the `IVersionVectorSyncService` by comparing the disconnected replica's version vector with a fully-synced server's version vector.

```csharp
using Ama.CRDT.Models;
using Ama.CRDT.Services.Journaling;
using Ama.CRDT.Services.Versioning;

// ... Inside your sync handler ...

// 1. You receive the client's current Version Vector state
var clientDvv = new DottedVersionVector(clientVersionVector, clientSeenExceptions);

// 2. You fetch the server's current Version Vector state for the document
var serverDvv = new DottedVersionVector(serverVersionVector, serverSeenExceptions);

// 3. Calculate what the client is missing from the server
var syncService = serviceProvider.GetRequiredService<IVersionVectorSyncService>();
var syncRequirement = syncService.CalculateRequirements(target: clientDvv, source: serverDvv);

// 4. Use the Journal Manager to fetch exactly those missing operations
var journalManager = serviceProvider.GetRequiredService<IJournalManager>();
var missingOperationsStream = journalManager.GetMissingOperationsAsync(syncRequirement);

var operationsToPushToClient = new List<CrdtOperation>();
await foreach (var journaledOp in missingOperationsStream)
{
    operationsToPushToClient.Add(journaledOp.Operation);
}

// 5. Send these operations back to the client as a CrdtPatch!
var patchForClient = new CrdtPatch(operationsToPushToClient);
```

By leveraging `ICrdtOperationJournal`, `IJournalManager`, and `IVersionVectorSyncService`, you can easily build highly resilient delta-sync protocols for distributed systems.
